6. 做得好的地方

6.1 日志驱动定位

支付和阅读器问题里，你经常提供 logcat、测试反馈、真实状态，例如 pay_verify、subscribeList、webLauncher callback、orderNum 等。这样能把 Claude 从猜测拉回证据链。

这是 Lereader 使用 Claude Code 中最高价值的模式：先给现象和日志，再让 Claude 追代码，而不是只描述“好像不对”。

6.2 文档优先意识已经形成

项目里已经形成了明确约束：

  • 阅读器先读 .claude/docs/reader/read_activity_analysis.md；
  • 支付先读 .claude/docs/payment/README.md；
  • 路由先读 .claude/docs/routing/notification_routing.md；
  • 实验先读 .claude/docs/platform/experiment-keys.md。

这能显著降低 Claude 读错入口、漏掉平行分支、重复探索旧问题的概率。

6.3 能把踩坑固化成规则

多个历史问题已经沉淀为长期规则：

  • 支付弹窗只有支付成功才 dismiss；
  • 验单失败/取消不能自动关闭弹窗；
  • Gson HashMap.class 数字会变 Double；
  • 提交前删除调试 Log.d；
  • SingleLiveEvent 重试要先设哨兵；
  • 异步转同步缓存要检查调用链时序依赖。

这说明你不是只解决一次问题，而是在让 Claude Code 逐步适配 Lereader 项目的真实风险。

6.4 验收意识强

你经常要求确认：

  • 成功分支是否关闭弹窗；
  • 失败分支是否弹 ConfirmPayDialog；
  • 取消/超时是否保留原状态；
  • SKU 为空时是否保护；
  • 覆盖安装旧缓存是否兼容；
  • scroll/page 模式是否一致。

这比单纯“修 bug”更接近工程验收。

7. 低效和重复劳动

7.1 同一长会话混入太多主题

高投入会话中出现过一个问题：最初是 ReadActivity pay_verify，中途又进入 UI 线条、支付、文档、提交流程等多个主题。

影响：

  • 上下文越来越肥；
  • Claude 容易带着旧任务假设处理新任务；
  • 后续 resume 难以判断当前目标；
  • 复盘统计难以归类。

建议：Lereader 中一个会话只承载一个主任务。支付、阅读器 UI、技术方案、commit/push 分开会话更稳。

7.2 关键约束经常中途补充

典型例子：

  • “MembershipRechargeRetainDialog 只有支付成功后才 dismiss”；
  • “验单失败后弹 ConfirmPayDialog”；
  • “三方支付回来后 orderNum 可能因为 subscribeList 刷新丢失”；
  • “支付 pay_type 需要通过 resolveCreateOrderPayType 转换”。

这些都是根约束，如果一开始没有写清楚，Claude 会按通用逻辑先给方案，之后再被你纠正。

建议：支付/阅读器任务一开始就给验收表。

7.3 技术方案粒度偏碎

feature 目录下多个 2026-05-26 的商店弹窗/Toast/埋点方案高度相邻：

  • store-popup-delayed-toast-plan
  • store-popup-sku-toast-plan
  • store-popup-toast-first-reason-plan
  • store-first-topup-log-plan
  • store-sub-read-free-month-log-plan

这些可能都合理，但它们容易重复分析同一批入口、埋点规则和弹窗链路。

建议：相邻小需求合成一个专题方案，内部按子需求拆章节，而不是每个点都新建独立方案。

7.4 .claude 文档体系开始有维护成本

.claude/docs 已经 61 个文件。继续增长后会出现两个问题：

  1. Claude 不知道优先读哪个；
  2. 旧文档和新文档可能重复或冲突。

建议：新增文档前先判断：更新旧文档还是新建文档。主索引必须同步维护。

7.5 工具系统建设会打断业务主线

Lereader 里 .claude 生态建设很有价值，但如果在业务 bug 中途改 skills、hooks、CLAUDE.md、memory，会拖长当前任务。

建议：业务任务中只记录“待沉淀项”，每周固定一次整理 .claude。

8. Lereader 提示词问题审查

8.1 “看一下”太宽泛

常见提示：

看一下阅读器内的 pay_verify 验单流程。

问题：Claude 不知道你要的是解释、找 bug、确认状态机、写文档，还是改代码。

建议写法：

模式：只读分析，不改代码。
目标：审查阅读器内 pay_verify 验单流程。
重点确认：三方支付返回后，成功/失败/取消分别如何影响 MembershipRechargeRetainDialog 和 ConfirmPayDialog。
输出：入口、状态机、orderNum 来源、弹窗动作、风险点。

8.2 验收标准要前置

支付类任务建议固定写：

场景期望行为
验单成功关闭支付弹窗，刷新权益
验单失败不关闭原弹窗，弹 ConfirmPayDialog
用户取消不自动 dismiss
orderNum 丢失说明丢失原因，检查 pending 兜底
SKU 未准备好不创建错误订单，不展示错误价格

8.3 阅读器任务要先声明模式

阅读器任务建议开头写：

模式：分析 / 方案 / 实现 / 验证。
范围：ReadActivity + ReadViewModel + 相关 Manager。
必须先读：.claude/docs/reader/read_activity_analysis.md。
不能做：不要顺手重构 ReadActivity，不要改无关支付/广告逻辑。

8.4 技术方案任务要把接口/实验/埋点放最前

适合 Lereader 的技术方案提示词：

模式：写技术方案，不实现。
需求：...
请优先列：
1. 是否需要后端接口改动；
2. 是否需要实验 key；
3. 是否需要新增/复用埋点；
4. 客户端涉及页面和入口；
5. 风险和验证方式。

9. Lereader 专项优化建议

9.1 每个任务用 6 行提示词开局

模式：只读分析 / 写方案 / 实现 / 验证。
目标：我要确认/修复/产出什么。
背景：已知现象、日志、测试反馈、业务原因。
约束：不能破坏什么、不能做什么、必须遵守什么。
输入：相关文件/文档/截图/log。
输出：我希望你按什么格式给结果。

9.2 支付和阅读器任务先做状态机

在动代码前先让 Claude 输出：

  • 触发入口；
  • 关键字段；
  • 状态变化；
  • UI 弹窗变化；
  • 成功/失败/取消分支；
  • 异步回调宿主存活检查。

9.3 技术方案合并相邻需求

商店弹窗、Toast、SKU、埋点这类相邻需求，建议先合并为一个专题：

商店弹窗体验优化技术方案
- 子需求 A：SKU 未准备 Toast
- 子需求 B：首充埋点
- 子需求 C：订阅免费月展示
- 子需求 D：延迟 Toast 原因

这样能减少重复读 StoreActivity、PaymentMethodDialog、埋点文件。

9.4 主文档优先，子文档补充

阅读器和支付已经有主文档。以后改相关模块时：

  1. 先更新主文档摘要；
  2. 再更新子文档细节；
  3. 避免只新建子文档但主索引不知道。

9.5 业务任务和 Claude 工具建设分离

建议节奏：

  • 工作日：专注业务任务，少改 .claude 基础设施；
  • 每周一次：统一整理 CLAUDE.md、skills、hooks、memory、docs 索引。

10. 最重要结论

Lereader 项目里，你已经把 Claude Code 用成了深度工程协作工具。主要提升空间不是“让 Claude 更聪明”，而是：

  1. 任务开始前把模式、目标、约束、验收标准说清楚；
  2. 支付/阅读器这类复杂链路先画状态机再改代码；
  3. 长会话减少混题，技术方案减少碎片化；
  4. .claude 文档继续沉淀，但要控制索引和维护成本。

如果只保留一个行动建议：

Lereader 每个任务开头先写 6 行：模式、目标、背景、约束、输入、输出。这样比继续增加新 skill 或 hook 更快减少返工。
